## Summary

Claude Code 2.1.143 made Shelly's native Bun SEA route unsafe as a default
again. The release can print a version and may draw the first TUI frame, but
foreground TUI/tool paths hit `Bus error` / `Segmentation fault` on-device.

The product decision after this update is:

- Default route: extracted `cli.js` from Claude's Bun SEA, executed by Shelly's
  bundled bionic Node.
- Native musl Bun SEA route: research / explicit opt-in only.
- Runtime updater: promote extracted Node route only after smoke checks.

## Observed Symptoms

Foreground native Claude route on Z Fold6:

```text
BashBus error
Bus error
Segmentation fault
```

The crashing command path involved Shelly's musl trampoline and Android linker:

```text
/system/bin/env -i ... SHELLY_MUSL_LD_PRELOAD=... /system/bin/linker64 ... shelly_musl_exec ...
```

Interpretation:

- This is not a Scouter bug.
- This is not caused by hooks or widget updates.
- `claude --version` is not a sufficient native smoke test.
- Native TUI/tool execution must remain opt-in until it passes longer
  real-device sessions.

## Packaging Change

Runtime extraction was out of sync with CI.

CI already handled Claude Code 2.1.133+ bundles where the `.bun` section no
longer has the old 4-byte little-endian size prefix before the JS body.
Runtime updater still assumed that prefix:

```js
const size = bunSection.readUInt32LE(startRel - 4);
const src = bunSection.subarray(startRel, startRel + size).toString('utf8');
```

For 2.1.143, the bytes before `// @bun` are marker-string tail bytes, not a
real size. Runtime updater now matches CI:

1. Find `file:///$bunfs/root/src/entrypoints/cli.js`.
2. Find `// @bun` after that marker.
3. Scan forward through text bytes.
4. Trim to the final Bun CJS wrapper close `})\n`.
5. Validate wrapper head and tail.
6. Patch tmpdir/browser bridge paths.
7. Lower `using` / `await using`.
8. Inject Shelly's Bun-on-Node polyfill.

Intentionally not shimmed:

- `Bun.listen`: Bun's TCP listener API is not equivalent to Node HTTP server.
  A wrong shim is worse than leaving it absent because it can hang or leak.
- `Bun.Terminal`: unknown behavior in Claude TUI path; do not fake until a
  concrete call site is captured.
- `Bun.Transpiler`: do not fake unless a non-TUI code path requires it.

## Background Updater Policy

Before `62c6fae9`, authenticated devices could run functional `--print` smoke
implicitly because credentials existed. That can spam background errors when
the extracted/Bun compatibility layer is broken.

Policy after this update:

- Default updater smoke: `node cli.js --version`.
- Functional smoke: only when `SHELLY_UPDATER_FUNCTIONAL_CHECK=1`.
- Native updater smoke/promotion: only when `SHELLY_UPDATER_NATIVE_CLAUDE=1`.

## Shell Launcher Regression

Follow-up device testing showed Claude Code could start but every Bash tool
call failed with exit code 1, and the terminal printed stale diagnostic lines:

```text
[DBG-A start]
[CKPT-id8]
claude-code_2-1-131_harness
```

Those strings are not in the current source. The likely root cause is a stale
device-side shell launcher/harness surviving APK upgrades.

Fix policy:

- CI builds `libshelly_shell.so`.
- `LibExtractor` must extract it as `shelly_shell`.
- `shelly_shell` must be in `ALWAYS_REFRESH`.
- Generated `.bashrc` points `SHELL=$SHELLY_LIB_DIR/shelly_shell`, then
  symlinks `$HOME/bin/bash` to it for tools that validate basename `bash`.
- Claude extracted/legacy Node routes must not inherit user `NODE_OPTIONS`,
  `BASH_ENV`, or `ENV` by default. User `NODE_OPTIONS` is opt-in via
  `SHELLY_CLAUDE_ALLOW_USER_NODE_OPTIONS=1`.

Follow-up device testing then showed a second stale-runtime variant:

```text
[shelly] claude: verified latest via extracted Bun cli.js (Node)
[DBG-A start]
[CKPT-id8]
2.1.131 (Claude Code)
```

`.bashrc_version` and `SHELL_OK` were correct, so the shell launcher fix was
active. The remaining issue was that
`~/.shelly-runtime/claude-extracted/version` could claim the latest release
while `current/.../cli.js --version` still executed an old 2.1.131 harness.

Fix policy:

- Treat `cli.js --version` as the source of truth, not the `version` marker.
- Reject known stale harness markers (`[DBG-A]`, `[CKPT-*]`,
  `claude-code_2-1-131_harness`) even if the command exits successfully.
- On mismatch, remove `claude-extracted/current` and `version`, write
  `.needs-repair`, and fall back to the APK extracted route in the same
  foreground `claude` launch.
- Background updater fast paths must also validate the current extracted
  runtime before logging `already on verified latest`.

## 2026-05-18 Follow-up: Bash Fixed, Paste Froze

Build `.bashrc_version` 148 fixed the obvious `claude --version` and direct
`$SHELL -lc 'echo SHELL_OK; pwd'` checks, but it introduced a worse foreground
regression: Claude could start, paste stopped responding, and Shelly could be
left in a state where relaunching the app failed.

Root causes identified in the 148 changes:

- The `Bun.spawn` object-form shim converted unknown `stdin` values to a pipe
  but did not write and close the payload. Claude tool calls could wait forever.
- The shim replaced `child.stdout` / `child.stderr` with Web streams, which can
  break consumers expecting Node streams.
- The `child_process.exec` wrapper dropped callbacks for the
  `exec(command, callback)` overload.
- `.shelly_paste_force_tui` was a global marker. If Claude crashed or the app
  died while the marker existed, every later terminal pane could be forced into
  the wrong paste path.
- `shelly_shell` preserved any non-empty `LD_PRELOAD`, including musl/Termux
  preload values that are wrong for bionic Bash.

Fix policy in `.bashrc_version` 149:

- Preserve `exec(command, callback)` callbacks.
- For object-form `Bun.spawn`, write supported `stdin` payloads and always end
  the pipe. When no stdin is supplied, default stdin to `ignore` instead of a
  Node pipe.
- Do not mutate child stdout/stderr streams.
- Validate the paste marker PID against the current shell PID; delete stale or
  malformed markers. Shelly's stream-backed PTY must propagate the real child
  PID into `TerminalSession.getShellPid()` so that validation is meaningful.
- Replace `LD_PRELOAD` unless it is exactly Shelly's bionic
  `libexec_wrapper.so`.
- Keep local LLM `llama-server` auto-install opt-in only, and timeout GitHub
  release lookup, so agent startup cannot look like a UI freeze.

If the promoted extracted runtime is stale, one launch may instead show:

```text
[shelly] claude: runtime extracted version mismatch (...); using APK extracted tier
[shelly] claude: APK extracted Bun cli.js (Node)
```

That is expected: the foreground session uses the bundled safe copy while the
next updater run repairs `~/.shelly-runtime/claude-extracted/current`.

Follow-up device testing confirmed `claude --version`, interactive TUI startup,
and a direct shell smoke all worked on 2.1.143, but Claude's internal Bash tool
still failed every command with `Exit code 1` and no stdout.

Root-cause policy after agent review:

- Treat direct `"$SHELL" -lc ...` success as only a shell-launcher smoke, not a
  Bash tool guarantee.
- The extracted Node route must patch Node `child_process` calls that pass
  `shell: true` so they use `/system/bin/sh` on Android.
- `exec()` / `execSync()` also need an explicit shell when the caller omits
  `shell`, because Node otherwise falls back to `/bin/sh`.
- The extracted Node route must support Bun's object-form `Bun.spawn({ cmd,
  cwd, env, stdin, stdout, stderr })`, because newer Claude Code builds can use
  that shape for tools.
- Bun-style `stdout` / `stderr` should be exposed as Web streams when possible;
  a raw Node `ChildProcess` is not a full Bun subprocess contract.
- `shelly_shell` must repair a minimal environment when Claude launches it from
  a sanitized tool env: `LD_PRELOAD`, `LD_LIBRARY_PATH`, `SHELLY_LIB_DIR`, and
  a usable `PATH`, including empty or reduced env values.
- Generated `.bashrc` must keep `BASH` aligned with the final
  `$HOME/bin/bash` shell path, not the earlier native-library launcher path.

Required on-device regression check:

```sh
cat ~/.bashrc_version
SHELLY_VERBOSE_CLI_TIER=1 claude --version
"$SHELL" -lc 'echo SHELL_OK; pwd'
claude
```

Inside Claude, ask for a Bash tool call such as:

```text
Bashテストを実行して結果だけ表示してください
```

Expected Bash tool command:

```sh
echo "Bash test: OK" && bash --version | head -1 && pwd
```

Native route must only be tested explicitly:

```sh
SHELLY_PREFER_NATIVE_CLAUDE=1 SHELLY_UPDATER_NATIVE_CLAUDE=1 SHELLY_VERBOSE_CLI_TIER=1 claude
```

Do not promote native as default unless all of these pass on-device:

- bare TUI launch
- login or already-authenticated startup
- workspace trust / first-run path
- Bash tool execution
- long-running session
- repeated tab open/close without crash loops

## 2026-05-18 Follow-up: `.bashrc_version` 150

Device testing of v149 narrowed the remaining 2.1.143 failure down to the
Claude-internal Bash tool: `"$SHELL" -lc ...`, `claude --version`, and the
interactive TUI all worked, while `Bash(echo test)` returned exit code 1 with
no stdout.

Fix policy in `.bashrc_version` 150:

- Treat `shell: "/bin/sh"`, `shell: "/bin/bash"`, `shell: "sh"`, and
  `shell: "bash"` the same as `shell: true`; route them to Shelly's generated
  shell instead of Android-missing `/bin/sh`.
- Repair child-process env for Claude tool calls: keep `PATH` usable and set
  `SHELL`, `BASH`, `SHELLY_LIB_DIR`, and `LD_LIBRARY_PATH` when Claude passes a
  reduced environment.
- Resolve bare `bash` / `sh` command launches to Shelly's shell wrapper.
- Make `shelly_shell` itself restore `SHELL` and `BASH` when launched from a
  sanitized environment, so nested Bash-tool subprocesses keep pointing at the
  launcher.
- Make the APK-bundled extracted `cli.js` include the same Bun/Node polyfill as
  runtime extraction. A stale runtime extracted tree that lacks the shell patch
  is invalidated and the APK extracted tier is used.

Future-update policy:

- A Claude Code update is not considered safe because `claude --version` or the
  TUI splash works. It must pass a tool-call canary that covers shell, file, git,
  search, and paste-sensitive paths.
- The goal is to keep upstream updates automatic, but gate promotion on these
  functional checks so a new release does not reach the visible `claude` command
  with broken tools.

## 2026-05-18 Follow-up: `.bashrc_version` 152

The update gate now treats an authenticated device as eligible for functional
Claude canaries by default. `--version` still runs first, but it is not enough to
mark the promoted runtime as healthy.

Functional canary behavior:

- The updater always smoke-tests `shelly_shell` under a reduced environment
  before promoting a Claude extracted runtime. This catches the class of
  failures where Claude's Bash tool cannot find Shelly's generated shell even
  though `claude --version` works.
- `claude --print` must return a literal sentinel.
- Claude must successfully use the Bash tool to run a literal sentinel command.
  The canary invokes Claude with explicit Bash tool availability and
  `bypassPermissions` so a healthy runtime is not rejected by interactive tool
  approval policy.
- A passing runtime writes `.shelly-functional-smoke-ok` inside the promoted
  `claude-extracted/<version>` package.
- `shelly-update-clis --check-only` returns "upgrade available" when the newest
  version is installed but the canary marker is missing, forcing a background
  revalidation instead of silently trusting a stale promotion.
- Automatic canaries require real Claude credentials. Shelly's generated
  `.claude.json` trust seed alone is not treated as auth. File-backed
  credentials and Claude env-token auth (`ANTHROPIC_AUTH_TOKEN`,
  `CLAUDE_CODE_OAUTH_TOKEN`, `ANTHROPIC_API_KEY`) are eligible.
- Auth, network, and tool-permission failures are diagnostic failures, not
  binary cooldown failures. They are temporarily suppressed for the same
  version/auth fingerprint to avoid retry loops until credentials or settings
  change.
- Extracted Node routes keep `LD_PRELOAD=$SHELLY_LIB_DIR/libexec_wrapper.so`
  so absolute `/bin/sh` / `/bin/bash` child process launches can still be
  redirected by Shelly's bionic exec wrapper.
- `shelly-doctor` reports the canary marker, recent runtime failure log tail,
  and recent canary suppressions.
- Manual trigger: `shelly-runtime-canary`.

This is intentionally narrow. The first goal is to stop repeating the
2.1.143-style failure where `claude --version` and TUI startup both worked but
tool execution was broken. Broader canaries for file, grep, git, paste, and
Scouter JSONL/hook behavior should be layered on top after the widget work is
back on track.

## Rule Of Thumb

When Claude Code updates:

1. Treat upstream native SEA changes as hostile until real-device TUI proves
   otherwise.
2. Keep extracted Node route as the default stability path.
3. Keep CI extraction and runtime extraction byte-for-byte conceptually aligned.
4. Do not add broad Bun shims. Add only APIs with known call sites, and prefer
   absent/diagnostic over wrong behavior.
5. Always bump `BASHRC_VERSION` when generated preload or shell routing changes.
